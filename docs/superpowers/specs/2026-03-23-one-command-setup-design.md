# One-Command Setup & First-Run Wizard

## Overview

A single SSH command that installs, configures, and starts not-ify on any Linux system with Docker. The SSH script handles infrastructure (containers, volumes, music folder), then directs the user to a web-based first-run wizard for account creation and service configuration.

**Target user:** Non-technical. Can SSH into their NAS but has never edited a compose file or YAML.

**Supported platforms:** Any Linux with Docker — QNAP, Synology, Raspberry Pi, VPS, desktop Linux.

## Entry Point

```bash
docker run --rm -it \
  -v /var/run/docker.sock:/var/run/docker.sock \
  -v /:/host:ro \
  ghcr.io/illtrick/not-ify:latest setup
```

- `--rm -it`: interactive, auto-cleanup
- Docker socket: lets the setup container manage host containers
- `/:/host:ro`: read-only host filesystem access for storage detection and folder browsing (the setup script reads only — writes go through Docker volume mounts)

The `setup` argument triggers the setup script (`/app/scripts/setup.sh`) instead of the normal server entrypoint.

## Part 1: SSH Setup Script

Lives inside the Docker image at `/app/scripts/setup.sh`. Runs inside a temporary container with host filesystem visibility.

### Step 1: Platform Detection

Reads `/host/proc/mounts` (host's mount table via the bind mount) to identify the platform:

| Platform | Signal | Default install dir |
|----------|--------|-------------------|
| QNAP | `/share/CACHEDEV*` mounts | `/share/CACHEDEV1_DATA/not-ify/` |
| Synology | `/volume*` mounts | `/volume1/docker/not-ify/` |
| Generic Linux | Everything else | `~/not-ify/` or `/opt/not-ify/` |

Confirms with the user: "Detected QNAP NAS. Install to /share/CACHEDEV1_DATA/not-ify/? [Y/change]"

### Step 2: Storage Detection & Music Folder Selection

Scans `/host/proc/mounts` for data filesystems (ext4, btrfs, xfs, ntfs — excludes tmpfs, proc, sysfs, devpts, overlay). Checks free space via `df`. Presents:

```
Where should not-ify store your music library?

Found these storage locations:
  [1] /share/CACHEDEV3_DATA/Media  (1.8 TB free)
  [2] /share/CACHEDEV1_DATA        (200 GB free)
  [3] Browse folders...
  [4] Enter path manually

Select [1-4]:
```

If user selects a location, asks for subfolder name (default: `not-ify-music`). Creates it if it doesn't exist.

If user selects "Browse folders..." — uses `whiptail --menu` (if available) to navigate directories interactively. Falls back to numbered directory listing if whiptail is not installed. User can create new folders inline.

If user selects "Enter path manually" — prompts for full path, validates it exists or offers to create it.

**Validation:** Checks the selected path is writable by creating and removing a test file.

### Step 3: Generate Configuration

Creates the install directory structure:

```
/share/CACHEDEV1_DATA/not-ify/
├── docker-compose.yml    (generated)
├── .env                  (generated, generic defaults)
├── .env.local            (generated, instance-specific: paths, generated keys)
├── config/               (created, bind-mounted as /app/config)
│   └── notify.db         (created on first app start)
└── slskd/                (created, for slskd config)
    └── slskd.yml         (generated with auto-generated API key)
```

**docker-compose.yml** — generated from a template inside the image. Contains:
- `not-ify` service: the app, ports, health check, bind mounts for config + music
- `slskd` service: Soulseek client, bind-mounted downloads directory shared with not-ify (host path: `<install-dir>/slskd-downloads`, mounted at `/app/slskd-downloads` in not-ify and `/app/downloads` in slskd)
- `gluetun` service (commented out): VPN sidecar, user can uncomment later
- `watchtower` service: auto-updates not-ify and slskd images

All paths reference `.env` / `.env.local` variables, not hardcoded values.

**.env** — committed-safe defaults:
```
PORT=3000
NODE_ENV=production
LOG_LEVEL=info
DLNA_ENABLED=true
```

**.env.local** — instance-specific (gitignored):
```
MUSIC_DIR=/share/CACHEDEV3_DATA/Media/not-ify-music
CONFIG_DIR=/share/CACHEDEV1_DATA/not-ify/config
SLSKD_API_KEY=<auto-generated-uuid>
SLSKD_DOWNLOADS_DIR=/share/CACHEDEV1_DATA/not-ify/slskd-downloads
```

**slskd.yml** — generated with the same auto-generated API key. Soulseek credentials left blank — entered through the web wizard later.

### Step 3b: Port Conflict Detection

Before starting containers, check if the required ports are available:
- Port 3000 (not-ify web UI)
- Port 5030 (slskd)

Use `docker` CLI to check for containers already bound to these ports, or `ss -tlnp` via `/host/proc/net/tcp`. If a conflict is found:

```
⚠️  Port 3000 is already in use by container "jellyfin"

  [1] Use a different port (e.g., 3001)
  [2] Stop the conflicting container
  [3] Quit

Select [1-3]:
```

If the user picks a different port, update `.env` accordingly.

### Step 4: Start Containers

```bash
cd <install-dir>
docker compose --env-file .env --env-file .env.local up -d
```

Waits for each container to become healthy (up to 60s per container). Shows progress:

```
Starting services...
  ✅ not-ify      (healthy, v1.5.2)
  ✅ slskd        (healthy)
  ⏳ watchtower   (starting...)
  ✅ watchtower   (running)
```

### Step 5: Health Verification

Checks each service endpoint from inside the setup container:
- not-ify: `GET http://<host-ip>:<port>/api/health`
- slskd: `GET http://<host-ip>:5030/api/v0/application` with API key header

If a service fails health check after 60s:
1. Shows the container logs (last 10 lines)
2. Offers: "[R]etry / [S]kip / [Q]uit"
3. On retry: restarts the container, re-checks
4. On skip: continues but warns the service won't be available

### Step 6: Print URL & Exit

Detects the host's LAN IP. Since we're inside a container, `hostname -I` returns the container IP, not the host. Instead, detect via:
1. Query Docker API through socket: `curl -s --unix-socket /var/run/docker.sock http://localhost/info | jq -r '.Swarm.NodeAddr // empty'`
2. Parse host's route table: `cat /host/proc/net/route` → find default gateway interface → read IP from `/host/proc/net/fib_trie`
3. Fallback: `ip route get 1.1.1.1 | grep -oP 'src \K[\d.]+'` from the host network namespace

Prints:

```
══════════════════════════════════════════════
  ✅ Not-ify is ready!

  Open in your browser:
  http://192.168.0.34:3000

  You'll create your account and connect
  your music services from there.
══════════════════════════════════════════════

To manage not-ify later:
  cd /share/CACHEDEV1_DATA/not-ify
  docker compose logs -f          # view logs
  docker compose restart           # restart
  docker compose down              # stop
  docker run --rm -it -v /var/run/docker.sock:/var/run/docker.sock ghcr.io/illtrick/not-ify:latest setup   # update
```

### Update Mode

If the script detects an existing installation (docker-compose.yml + config/ dir exist at the install path), it enters update mode:

**Version detection:**
- Current version: `curl -s http://localhost:<port>/api/health | jq -r .version` (if app is running), or parse the image tag from the running container via Docker API
- Available version: the setup container IS the latest image, so its bundled version is the target. Read from `/app/package.json` inside the setup container.

```
Existing not-ify installation found at /share/CACHEDEV1_DATA/not-ify/
  Running: v1.5.1  →  Available: v1.5.2

  [U]pdate containers (keeps all data)
  [R]econfigure (re-run setup, keeps data)
  [F]resh install (⚠️ see below)
  [Q]uit
```

**Update:** pulls latest images, recreates containers, runs health checks. Config, DB, music untouched.

**Reconfigure:** re-runs storage detection and folder selection, regenerates compose file. Existing config/DB preserved unless user explicitly chooses otherwise.

**Fresh install:** requires explicit confirmation with clear description of what's lost:
```
⚠️  Fresh install will DELETE:
  - Your user account(s)
  - Listening history (scrobbles)
  - All saved credentials (Last.fm, Real-Debrid, VPN, Soulseek)

  Your music files will NOT be deleted.

  Type "RESET" to confirm:
```

## Part 2: Web First-Run Wizard

### Trigger

The app detects first-run state: `users` table is empty in the DB. When this is true, ALL routes except `/api/health` and `/api/setup/*` redirect to the setup wizard.

**Redirect mechanism:** Express middleware checks `db.getUsers().length === 0` on every request (cached after first check, invalidated when a user is created). API routes return `{ error: 'setup_required', setupUrl: '/setup' }` with 403 status. The client checks for this response and redirects to the setup wizard SPA route.

### Setup API Endpoints

```
GET  /api/setup/status          — returns { needsSetup: bool, completedSteps: [] }
POST /api/setup/account         — { displayName } → creates admin user, returns { userId, isAdmin }
GET  /api/setup/library         — returns { musicDir, exists, writable, freeSpace }
PUT  /api/setup/library         — { musicDir } → updates music library path
POST /api/setup/service/:name   — { credentials } → saves + tests a service (lastfm, realdebrid, vpn, soulseek)
GET  /api/setup/services        — returns status of all services { name, configured, connected, error }
POST /api/setup/complete        — marks setup as done, redirects to app
```

The `/api/setup/*` routes do NOT require authentication (no user exists yet). After `POST /api/setup/complete`, these routes return 404.

### Blocking Steps (required)

**Step 1: Create Admin Account**

```
Welcome to Not-ify!

Let's set up your account.

  Display name: [Nathan          ]

  [Create Account]
```

No password — single-household LAN app. The first user created is automatically admin. Future users can be added from Settings.

**Step 2: Confirm Music Library**

```
Your music library is at:
  /share/CACHEDEV3_DATA/Media/not-ify-music

  📁 Empty (ready for music)
  💾 1.8 TB free

  [Looks good] [Change location]
```

Shows the path configured by the SSH setup script. Validates it's readable/writable. If the user wants to change it, opens the FolderBrowser component (already built in Settings — `packages/client/src/components/FolderBrowser.jsx`). This component uses the `/api/library-config/browse` endpoint which lists directories visible to the container.

### Optional Steps (skippable)

Each step has a "Skip for now" link and a "Test Connection" button.

**Step 3: Last.fm**
- What it does: "Track your listening history and get personalized recommendations"
- Fields: API Key, Shared Secret
- Link to get API key
- Test: authenticates and shows username

**Step 4: Real-Debrid**
- What it does: "Download higher quality versions of your music"
- Fields: API Token
- Test: shows account status + expiry

**Step 5: VPN (PIA)**
- What it does: "Route downloads through a VPN for privacy"
- Fields: Username, Password, Region dropdown
- Test: stores credentials in DB, then calls `POST /api/setup/service/vpn` which:
  1. Checks if gluetun container exists (it's generated commented-out in the compose template)
  2. If not running: uncomments gluetun in docker-compose.yml via the Docker socket, restarts the stack
  3. Writes PIA credentials to gluetun's config
  4. Waits for gluetun health check, returns VPN IP
- **If gluetun fails to start:** shows error with "Skip for now" option. VPN is not required.

**Step 6: Soulseek**
- What it does: "Find rare and lossless music from other collectors"
- Fields: Username, Password
- Test: calls `POST /api/setup/service/soulseek` which:
  1. Writes credentials to `slskd.yml` (via shared config volume)
  2. Restarts slskd container via Docker socket
  3. Polls slskd health API with the auto-generated API key
  4. Returns connection status (Connected/LoggedIn/Failed)

### Final Step

**Service Dashboard**

```
You're all set! Here's your service status:

  🟢 Music Library    /share/.../not-ify-music (1.8 TB free)
  🟢 Last.fm          Connected as smaerDmuipO
  🟢 Real-Debrid      Premium (expires 2026-10-27)
  🟡 VPN              Not configured — Skip
  🟡 Soulseek         Not configured — Skip

  [Start Listening]
```

"Start Listening" dismisses the wizard permanently. The app enters normal mode. Unconfigured services show gentle warnings in their respective Settings sections.

### Re-entry

The wizard is NOT shown again after dismissal. Users configure services through Settings. However, if the DB is reset (fresh install via setup script), the wizard triggers again.

## Config Architecture

### Three-Layer Config

| Layer | File | Contains | In git repo |
|-------|------|----------|------------|
| Defaults | `.env` (generated on host) | Feature flags, log level, port | No — but `.env.example` in repo serves as template |
| Instance | `.env.local` (generated on host) | Paths, generated keys | No (gitignored) |
| User data | `config/notify.db` (bind mount) | Credentials, scrobbles, library | No |

### What Goes Where

| Value | Set by | Stored in | Survives update |
|-------|--------|-----------|-----------------|
| PORT | Setup script | `.env` | Yes |
| NODE_ENV | Setup script | `.env` | Yes |
| MUSIC_DIR | User (setup script) | `.env.local` | Yes |
| CONFIG_DIR | Setup script | `.env.local` | Yes |
| SLSKD_API_KEY | Auto-generated | `.env.local` + `slskd.yml` | Yes |
| DLNA_ENABLED | Setup script | `.env` | Yes |
| Last.fm/RD/VPN/Soulseek creds | Web wizard | SQLite DB | Yes |
| User accounts | Web wizard | SQLite DB | Yes |
| Scrobble history | Last.fm sync | SQLite DB | Yes |

## Database Integration

### Changes from current state

The current DB seeds three hardcoded users (`default`, `nathan`, `sarah`) on every fresh initialization and auto-promotes the first non-default user to admin. This must change to support the first-run wizard.

### What changes

**1. Remove hardcoded user seeding** (db.js lines 186-190)

Remove:
```javascript
upsertUser.run('default', 'Default');
upsertUser.run('nathan', 'Nathan');
upsertUser.run('sarah', 'Sarah');
```

On a fresh DB, the `users` table is now **empty**. The first-run wizard creates the first user.

**2. Remove auto-admin promotion** (db.js lines 180-184)

Remove the logic that promotes the first non-default user. Instead, the `POST /api/setup/account` endpoint explicitly creates the user with `role = 'admin'`.

**3. Add user creation functions to db.js**

```javascript
function createUser(id, displayName, role = 'user') {
  getDb().prepare('INSERT INTO users (id, display_name, role) VALUES (?, ?, ?)').run(id, displayName, role);
  return { id, displayName, role };
}

function getUserCount() {
  return getDb().prepare('SELECT COUNT(*) as count FROM users').get().count;
}

function getDefaultUserId() {
  const row = getDb().prepare("SELECT id FROM users WHERE id != 'default' ORDER BY created_at ASC LIMIT 1").get();
  return row?.id || null;
}
```

**4. Replace `default` user fallback in user middleware**

```javascript
// Old:
req.userId = req.headers['x-user-id'] || 'default';

// New:
req.userId = req.headers['x-user-id'] || db.getDefaultUserId();
if (!req.userId) {
  // No users exist — only setup routes allowed
  if (req.path.startsWith('/api/setup/') || req.path === '/api/health') {
    return next();
  }
  return res.status(403).json({ error: 'setup_required', setupUrl: '/setup' });
}
```

**5. Add `setup_complete` flag**

`POST /api/setup/complete` writes to `global_settings`:
```javascript
db.setGlobalSetting('setup_complete', true);
```

The setup middleware checks this on startup (cached in memory, not queried per-request):
```javascript
let setupComplete = db.getGlobalSetting('setup_complete') || db.getUserCount() > 0;
```

This means: if users exist (from old DB or wizard), setup is complete. The flag is a belt-and-suspenders check.

### What does NOT change

- All table schemas (no ALTER TABLE, no migrations)
- `global_settings`, `lastfm_config`, `tracks`, `scrobbles`, `artist_affinity`, `mb_cache` tables
- `isAdmin()` function — still reads role column
- Admin guard middleware — unchanged
- Legacy JSON migration — unchanged

### Backward compatibility

| Scenario | Behavior |
|----------|----------|
| **Existing DB with `nathan`/`sarah`** | Users table not empty → wizard doesn't trigger → app works normally |
| **Existing DB, `default` user referenced** | `getDefaultUserId()` skips `default`, returns first real user |
| **Fresh DB** | Empty users table → setup middleware redirects to wizard → user creates account |
| **DB with users but no `setup_complete` flag** | `getUserCount() > 0` catches this → no wizard |

### Schema (unchanged)

No new tables needed. The `users` table already has `id`, `display_name`, `role`, `created_at`. The `global_settings` table already exists for the `setup_complete` flag. All setup API endpoints use existing tables.

## Docker Image Changes

### Entrypoint Modification

The Dockerfile's entrypoint needs to handle the `setup` argument:

```dockerfile
COPY scripts/setup.sh /app/scripts/setup.sh
COPY scripts/docker-compose.template.yml /app/scripts/docker-compose.template.yml

ENTRYPOINT ["/app/scripts/entrypoint.sh"]
CMD ["server"]
```

`entrypoint.sh`:
```bash
#!/bin/bash
if [ "$1" = "setup" ]; then
  shift
  exec /app/scripts/setup.sh "$@"
else
  exec node packages/server/src/index.js "$@"
fi
```

### Dependencies Added to Image

- `bash` (entrypoint and setup script — dash/sh is insufficient)
- `whiptail` (interactive TUI for folder browsing) — `whiptail` package on Debian. Always installed, not optional — the fallback to numbered menus is for non-interactive terminals only.
- Docker CLI + Compose plugin — `docker-ce-cli` and `docker-compose-plugin` packages. Required to manage host containers via the Docker socket mount.
- `jq` (for parsing Docker API responses and health check JSON)

These add ~30MB to the image. Acceptable given the setup UX improvement.

## Security Considerations

- **Docker socket mount:** Required for setup script to manage host containers. Standard practice (Portainer, Watchtower). User must trust the image source (GHCR under your control).
- **Host filesystem read-only:** `/:/host:ro` gives read access for storage detection. No writes through this mount — all writes go through Docker volume mounts on specific directories.
- **Auto-generated API key:** SLSKD_API_KEY is a UUID generated at install time. Never hardcoded, never committed.
- **No default users:** First-run wizard creates the admin. No baked-in accounts.
- **Credentials in DB only:** Never written to env files or compose files.

## Prerequisites

The setup command requires Docker installed on the host. If Docker is not present, the `docker run` command itself fails with a clear error. The README/landing page should include:

```
# Prerequisites: Docker must be installed
# QNAP: Install "Container Station" from App Center
# Synology: Install "Container Manager" from Package Center
# Linux: curl -fsSL https://get.docker.com | sh
```

Additionally, Docker Compose v2 must be available. The setup script checks for `docker compose version` early and exits with a helpful message if missing. Most modern Docker installations (23.0+) include the Compose plugin.

## Implementation Notes

- **UUID generation:** `SLSKD_API_KEY` is generated via `cat /proc/sys/kernel/random/uuid` (available on all Linux kernels). Fallback: `head -c 32 /dev/urandom | xxd -p`.
- **Docker Compose v2 detection:** `docker compose version 2>/dev/null` — if it fails, check for legacy `docker-compose` binary and use that. If neither exists, exit with install instructions.
- **TERM/TTY for whiptail:** The `-it` flags on the `docker run` command attach the host terminal. Whiptail reads `TERM`, `LINES`, `COLUMNS` from the environment. Docker passes these through automatically with `-it`. The setup script should verify `[ -t 0 ]` (stdin is a terminal) and fall back to non-interactive mode if not (e.g., piped input).
- **Compose file manipulation for VPN:** The setup script generates the compose file with gluetun commented out. The web wizard's VPN setup step calls a server endpoint that uses `sed` or a template engine to uncomment the gluetun service and restart via `docker compose up -d`. This requires the not-ify container to also have Docker socket access (already mounted for other management tasks).

## Out of Scope

- **Production deployment** — this is staging/personal use for now
- **Multi-node / clustering** — single host only
- **SSL/HTTPS** — reverse proxy is the user's responsibility
- **Backup/restore** — the config dir is the backup unit; user can copy it
- **Password authentication** — LAN-only app, added later if needed
