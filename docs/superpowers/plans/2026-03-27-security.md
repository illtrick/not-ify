### Task 1: Fix path traversal in library-config browse endpoint (S1)

**Severity:** CRITICAL

**Files:**
- Modify: `packages/server/src/api/library-config.js:67-68`
- Create: `packages/server/__tests__/api/library-config-traversal.test.js`

**Context:** The `/api/library-config/browse` endpoint resolves `req.query.path` with `path.resolve()` but never validates the result. An attacker can request `?path=../../etc` to browse any directory on the filesystem. The `/api/library-config/files-count` endpoint at line 88-89 has the same issue. The POST `/` endpoint at line 35 and POST `/migrate` at lines 128-134 also resolve user paths without restriction, but those are write-path-validated (stat check, writable check) and are admin-only setup endpoints. The browse endpoint is the primary risk since it returns directory listings.

The fix adds a helper that validates resolved paths start with an allowed root. In Docker, the allowed roots are `/app/music` and `/`. In non-Docker, we restrict to the music dir and common mount points. For the browse endpoint specifically, since its purpose IS to browse the filesystem (it's used during initial setup to pick a music directory), the appropriate fix is to ensure the endpoint is only accessible during setup (not after), and to deny access to sensitive system paths.

Actually, re-reading the code: this is a filesystem browser for the setup wizard. The intended use is to let the admin pick any directory. The real fix is to ensure it's behind the setup middleware (only accessible before setup completes or by admins). Let's validate that it rejects path traversal sequences and can't escape to sensitive directories.

In `packages/server/src/api/library-config.js`, add a validation helper after the imports (after line 5):

```javascript
// Sensitive directories that the browse endpoint should never expose
const BLOCKED_PREFIXES = ['/proc', '/sys', '/dev', '/run', '/var/run'];
const BLOCKED_EXACT = new Set(['/etc', '/etc/shadow', '/etc/passwd']);

function validateBrowsePath(requestedPath) {
  if (!requestedPath || typeof requestedPath !== 'string') {
    return { valid: false, error: 'Invalid path' };
  }
  // Reject null bytes (poison byte attack)
  if (requestedPath.includes('\0')) {
    return { valid: false, error: 'Invalid path: contains null bytes' };
  }
  const resolved = path.resolve(requestedPath);
  // Block sensitive system directories
  for (const prefix of BLOCKED_PREFIXES) {
    if (resolved === prefix || resolved.startsWith(prefix + '/')) {
      return { valid: false, error: `Access denied: ${resolved}` };
    }
  }
  if (BLOCKED_EXACT.has(resolved)) {
    return { valid: false, error: `Access denied: ${resolved}` };
  }
  return { valid: true, resolved };
}
```

### Task 2: Fix command injection in setup.js df command (S2)

**Severity:** CRITICAL

**Files:**
- Modify: `packages/server/src/api/setup.js:82-88`
- Create: `packages/server/__tests__/api/setup-injection.test.js`

**Context:** Line 87 uses `execSync(\`df -k "${musicDir}" 2>/dev/null | tail -1\`)` which interpolates `musicDir` into a shell command. A crafted `musicDir` like `/app/music"; cat /etc/passwd; echo "` would execute arbitrary commands. Fix: use `execFileSync('df', ['-k', musicDir])` which passes arguments as an array, bypassing the shell entirely.

The current code:

### Task 4: Fix heredoc injection in container-manager writeSlskdApiKeyConfig (S4)

**Severity:** CRITICAL

**Files:**
- Modify: `packages/server/src/services/container-manager.js:230-254`
- Create: `packages/server/__tests__/services/container-manager-injection.test.js`

**Context:** Line 237 constructs a shell command with a heredoc that includes the `apiKey` value:

```javascript
const cmd = `cat > /app/slskd.yml << 'CFGEOF'\n${configYml}CFGEOF`;
```

where `configYml` includes `key: ${apiKey}`. If `apiKey` contains `\nCFGEOF\n<malicious commands>`, the heredoc terminates early and the rest executes as shell commands inside the container.

Fix: Instead of using a heredoc via `sh -c`, write the config content directly using the Docker API's tar upload to write a file, or use a two-step approach: first create the file content, then write it via the exec API's stdin. The simplest safe approach is to use the Docker API to write the file content directly by sending it as stdin to `tee`.

Actually, the cleanest fix: construct the YAML content in Node.js, then use the Docker exec API to run `tee /app/slskd.yml` and pipe the content via stdin. But the Docker exec API's stdin handling is complex. A simpler approach: validate the API key doesn't contain newlines or YAML-special characters, then use a single `echo` with the exec Cmd array (no shell interpolation).

Best approach: Use `Cmd: ['sh', '-c', 'cat > /app/slskd.yml']` with the content sent via stdin attachment, or use multiple exec commands. The simplest safe fix is to pass each line as a separate `printf` argument array:

```javascript
const Cmd = ['sh', '-c', 'printf "%s" "$1" > /app/slskd.yml', '_', configYml];
```

This puts `configYml` into `$1` as a positional parameter, avoiding any heredoc or interpolation issues.

```javascript
'use strict';

/**
 * Simple in-memory rate limiter using a sliding window counter.
 * @param {object} opts
 * @param {number} opts.windowMs — time window in milliseconds (default: 60000)
 * @param {number} opts.max — max requests per window per IP (default: 30)
 * @returns {Function} Express middleware
 */
function createRateLimiter({ windowMs = 60000, max = 30 } = {}) {
  const hits = new Map(); // ip -> { count, resetTime }

  // Periodically clean up expired entries to prevent memory leak
  const cleanup = setInterval(() => {
    const now = Date.now();
    for (const [ip, entry] of hits) {
      if (now > entry.resetTime) hits.delete(ip);
    }
  }, windowMs * 2);

  // Don't keep the process alive just for cleanup
  if (cleanup.unref) cleanup.unref();

  function middleware(req, res, next) {
    const ip = req.ip || '0.0.0.0';
    const now = Date.now();

    let entry = hits.get(ip);
    if (!entry || now > entry.resetTime) {
      entry = { count: 0, resetTime: now + windowMs };
      hits.set(ip, entry);
    }

    entry.count++;

    if (entry.count > max) {
      const retryAfter = Math.ceil((entry.resetTime - now) / 1000);
      res.set?.('Retry-After', String(retryAfter));
      return res.status(429).json({
        error: 'Too many requests. Please try again later.',
        retryAfter,
      });
    }

    next();
  }

  return middleware;
}

module.exports = createRateLimiter;
```
